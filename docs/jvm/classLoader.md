## 1.类加载过程

Java 中的类加载过程是一个复杂而精细的步骤,它负责将类的字节码文件(.class 文件)加载到 Java 虚拟机(JVM)中,并为执行做准备。这个过程主要分为五个阶段:**加载(Loading)、验证(Verification)、准备(Preparation)、解析(Resolution)、初始化(Initialization)**。类加载是一个按需进行的过程,即只有当类被首次主动使用时,才会触发类的加载过程。这个“主动使用”包括但不限于创建类的实例、访问类的静态成员、子类初始化、反射调用类、初始化时指定的父类尚未初始化等。

### 1.1 加载阶段

加载阶段是类加载过程的第一个阶段,在这个阶段,类加载器(ClassLoader)通过查找类的二进制数据(通常是.class 文件),读取这些数据到 JVM 中,并为类在方法区(或称永久代/元空间)中创建一个表示它的 Class 对象。加载过程可以细分为以下几个子步骤:

- 查找并识别类的二进制数据:类加载器负责查找类的二进制数据。类加载器首先检查该类是否已经被加载过,如果已经存在对应的 Class 对象,则直接返回,避免重复加载。JVM 提供了三种基本的类加载器:启动类加载器(Bootstrap ClassLoader)、扩展类加载器(Extension ClassLoader)和系统类加载器(Application ClassLoader),它们共同构成了类加载器的层次结构。
- 读取类的二进制数据:对于从文件系统加载的情况,类加载器会读取.class 文件的内容。.class 文件是 Java 源代码经过编译后的二进制格式,包含了类的结构信息、常量池、方法代码等。除了从本地文件系统读取,类还可以从网络、数据库、运行时生成的字节码等多种来源获取。
- 类的字节流解析:JVM 需要将读取到的二进制数据解析成内部的数据结构,这个过程涉及将类的结构信息(如类名、父类名、接口、字段、方法等)转换为 JVM 能够理解并使用的数据结构。
- 创建 Class 对象:在解析完类的二进制数据后,JVM 会在方法区中为该类创建一个对应的 Class 对象。这个 Class 对象是所有该类实例的模板,包含有关类的元数据信息,如类的字段、方法、构造函数以及父类等。Class 对象也是 Java 反射的基础,通过它可以在运行时获取类的信息或创建类的实例。
- 分配内存:为新加载的类在方法区内分配内存,存储类的结构信息,包括类的版本、字段、方法、接口、常量池等。

加载阶段结束后,类的信息就已经被 JVM 所识别,但此时类还没有被完全验证,也没有进行静态变量的初始化。接下来的验证、准备、解析和初始化阶段将按顺序进行,最终使类达到可执行状态。

### 1.2 验证阶段

验证阶段确保加载的类信息是否符合 JVM 规范,防止恶意代码损害 JVM 的安全。这包括文件格式验证、元数据验证、字节码验证和符号引用验证等多个阶段:

- 文件格式验证:确保 Class 文件的字节流符合《Java 虚拟机规范》的定义,比如魔数、版本号、常量池入口的有效性等。这一步是最低层次的验证,用于检验 Class 文件的基本结构是否完整且合法。例如验证魔数是否是 CAFEBABE,主次版本号是否在当前虚拟机接受范围内,常量池中的常量类型和数量是否符合规范等。
- 元数据验证:对字节码的语义数据进行校验,保证类的元数据信息符合 Java 语言规范,如类是否有父类(除了 java.lang.Object)、类的方法和字段是否与父类相匹配、方法的描述符是否合法等。例如检查类的继承关系、抽象类是否有抽象方法的实现、方法的重写是否合法等。
- 字节码验证:这是验证阶段中最复杂的一个步骤,它通过数据流分析和控制流分析来确定程序语义的正确性,确保字节码指令不会危害虚拟机的安全。例如验证操作数栈和局部变量表的类型是否匹配、跳转指令的目标位置是否有效、方法调用的参数类型是否正确、异常处理表的合理性等。
- 符号引用验证:发生在解析阶段,主要针对类自身以外的信息进行校验,如常量池中的类或接口名、字段名、方法名的符号引用是否能在其他已加载的类中找到对应的实体。例如验证类、字段、方法的访问性,确保可以在当前类中正确引用；检查类继承的层次结构,确保不违反单一继承原则等。

通过这四个层次的验证,Java 虚拟机确保了加载的类是安全的、结构正确的,为后续的类准备、解析和初始化阶段打下了基础。如果在验证过程中发现任何问题,Java 虚拟机将会抛出相应的验证错误,阻止该类的加载。

### 1.3 准备阶段

准备阶段是 Java 类加载过程中的第三个阶段,在验证阶段之后进行。这个阶段的主要任务是为类的静态变量分配内存,并为其设置默认的初始值。需要注意的是,这个阶段不会执行静态变量的初始化代码(即静态初始化块中的赋值语句),而仅仅是为静态变量分配内存并赋予默认值:

- 静态变量分配内存:为类变量(static 修饰的变量)分配内存空间。这里的类变量指的是在类中声明且被 static 关键字修饰的变量,它们属于类本身,而不属于类的某个实例。分配的内存区域通常是方法区(Java 8 之前的 HotSpot 虚拟机)或元空间(Java 8 及之后的 HotSpot 虚拟机)。
- 默认值赋值:对于基本数据类型的静态变量,如 int、char 等,会赋予 Java 语言规范中规定的默认值。例如,int 类型的默认值为 0,boolean 类型的默认值为 false。对于引用类型的静态变量,无论是对象引用还是数组引用,都将被初始化为 null。

:::danger
注意:如果静态变量被 final 修饰,并且在声明时被直接赋值(即编译时常量),那么这个变量的值将在编译阶段就放入常量池中,准备阶段会直接从常量池中读取其值,而不再赋予默认值后再进行初始化。
:::

准备阶段的目的是为类变量分配必要的内存空间,确保这些变量在类的任何实例创建之前已经具备了存储的空间和默认的值。这一步是类实例化之前的必要准备,为接下来的解析和初始化阶段打下基础。

### 1.4 解析阶段

解析阶段是 Java 类加载过程中的第四个阶段,发生在准备阶段之后,初始化阶段之前。这个阶段主要是将常量池中的符号引用转换为直接引用的过程,以便于后续的访问操作能够快速定位到具体的内存地址。解析阶段可以延迟到第一次使用时才进行,这种策略称为“延迟解析”或“懒解析”。解析主要针对常量池中的符号引用,包括以下几类:

- 类或接口的符号引用:解析为该类或接口在方法区的直接引用。
- 字段的符号引用:解析为对应字段在类或接口的内存布局中的直接引用。
- 方法的符号引用:解析为对应方法在类或接口的内存布局中的直接引用,包括静态方法和实例方法。
- 接口方法的符号引用:解析为对应接口方法在接口内存布局中的直接引用。

:::tip
直接引用是指直接指向目标的指针、相对偏移量或者能间接定位到目标的句柄。相比于符号引用,直接引用是更贴近于虚拟机内存布局的引用,它让 JVM 能够快速定位到实际的类、方法、字段等信息,而无需再次解析
:::

解析过程不一定在类加载时立即进行,JVM 可以自由选择何时进行解析,甚至可以等到首次使用时才进行。这种设计是为了支持动态绑定,即方法调用的接收者不确定时,可以在运行时决定调用哪个方法实现。解析阶段是 Java 类加载过程中的一个重要步骤,它确保了类中的符号引用能够转换为可直接访问的内存地址,提高了程序运行时的效率。通过延迟解析策略,Java 虚拟机在保证灵活性的同时,也优化了资源的使用。

### 1.5 初始化阶段

初始化阶段是 Java 类加载过程中的最后一个阶段,也是最为关键的一步,它负责执行类的初始化代码,为类的静态变量赋予程序员指定的初始值,并执行静态初始化块中的代码。初始化阶段标志着类加载过程的完成,使得类可以被 JVM 正确地使用。类的初始化是在其主动使用时发生的,以下情况会触发类的初始化:

- 创建类的实例:通过 new 关键字实例化对象时。
- 访问类或接口的静态变量或静态方法:访问时该变量或方法必须是类的静态成员,且不为编译期常量。
- 反射调用:通过 Class.forName()方法显式加载类时。
- 初始化子类:如果子类被初始化,其父类也会被初始化(但接口的实现类初始化不会引起接口的初始化)。
- JVM 启动时指定的主类:执行 java 命令指定的主类会首先被初始化。
- 使用动态语言支持时:如使用 invokedynamic 指令。

类的初始化阶段执行过程如下:

- 执行类的静态初始化代码:包括静态字段的显式初始化赋值以及静态初始化块中的代码,这些代码按照在源文件中出现的顺序依次执行。
- 父类初始化:在初始化一个类之前,如果其父类还未被初始化,则先初始化父类(不包括接口)。
- 锁机制:为了线程安全,JVM 在初始化一个类时会采用加锁机制,确保类只被初始化一次。这意味着即使多个线程同时到达初始化阶段,也只会有一个线程执行初始化,其他线程等待直至初始化完成。

:::danger

- 接口初始化:接口不会被初始化,除非它包含有初始化代码,这种情况很少见,通常是因为接口中声明了静态初始化块或非常量的静态字段。
- 静态字段的初始化顺序:静态字段的初始化按照它们在类中声明的顺序进行,即使这些字段分布在不同的代码块中。
- 编译期常量:如果一个静态字段是编译期常量(即 final 修饰且初始化为编译时可知的值),那么这个字段的赋值不会引起类的初始化,因为这个值会被直接嵌入到使用它的类或方法的字节码中。

:::
初始化阶段是类加载过程的终点,它确保了类的静态成员具有正确的初始状态,是类能够被正确使用的前提。通过严格的初始化规则和线程安全的保障,Java 确保了类加载过程的稳定性和高效性。

## 2.类加载器(Class Loader)

类加载器(Class Loader)是 Java 运行时环境的一个核心组件,负责动态地将类或接口的字节码文件(通常是.class 文件)加载到 Java 虚拟机(JVM)的内存中,并创建对应的 java.lang.Class 对象。这个过程不仅仅是简单地读取字节码文件,还包括验证类的正确性、准备类的静态成员、解析类中的符号引用等步骤,最终使得类能够在 JVM 中被识别和执行。根据类加载器的层次结构,主要包括以下四种类型(除了启动类加载器外其他类加载器都被称为自定义加载器,自定义加载器都直接或间接继承自 ClassLoader,使用 Java 语言实现):

- 启动类加载器(Bootstrap ClassLoader):负责加载 Java 的核心库(如 rt.jar),它是由 C++实现的,是所有类加载器的父加载器,但是它本身并没有父加载器。
- 扩展类加载器(Extension ClassLoader):负责加载 Java 扩展目录(JAVA_HOME/jre/lib/ext)下的 jar 包或类库。
- 应用程序类加载器(Application ClassLoader):也称为系统类加载器,负责加载用户类路径(ClassPath)上指定的类库,是平时编写 Java 代码中默认使用的类加载器。
- 自定义类加载器(Custom ClassLoader):用户可以根据需求定制类加载器,用于加载特定目录、网络或其他来源的类文件,是 Java 提供的可扩展机制,允许开发者自定义类的加载逻辑。

### 2.1 类加载器基本特征

类加载器在 Java 中具有三个基本特征,它们分别是:

- 双亲委派模型(Parent Delegation Model):当类加载器收到加载类的请求时,它并不会立即尝试自己去加载这个类,而是将加载任务委托给其父类加载器去完成。这个过程会一直向上委托,直到最高层的启动类加载器(Bootstrap ClassLoader)。如果父加载器无法完成加载任务(例如找不到对应的类),则子加载器会尝试自己加载。这一机制确保了类的唯一性和安全性,比如防止基础类被篡改。在实际开发中,并非所有类加载器都严格遵守这一模型,有些特殊的场景或框架为了实现特定功能,会打破双亲委派模型,最典型的例子是 Web 容器如 Tomcat 中的类加载器设计。
- 可见性(Visibility):子类加载器可以访问父类加载器所加载的类,但是父类加载器却无法看到子类加载器加载的类。这样的设计是为了维持类加载器之间的隔离性,使得不同层级的加载器可以负责不同范围的类,避免类之间的混乱和冲突。
- 单一性(Uniqueness):由于父类加载器加载的类对于子类加载器是可见的,因此父类加载器中加载过的类型不会在子加载器中重复加载。这意味着对于任意一个给定的类,无论通过哪个加载器加载(只要该加载器在双亲委派链上,并且有父加载器已经加载了该类),最终得到的都是同一个 Class 对象。但是,如果两个加载器之间不存在父子关系,并且都尝试加载同一个类,那么可能会导致各自加载出不同的类实例,因为它们之间相互不可见。

## 4.双亲委派机制

双亲委派机制(Parent Delegation Mechanism)是 Java 类加载机制中的一个核心原则,旨在确保类加载的安全性和一致性。该机制定义了类加载器加载类时的一种委托行为模式。在 Java 的类加载器实现中,每个类加载器都有一个父类加载器的引用,除了启动类加载器(该加载器由 C++编写,没有父加载器)。这种父子关系构成了类加载器的层次结构,使得双亲委派得以实现。具体流程:当一个类加载器接收到加载类的请求时,并不直接尝试加载这个类,而是将加载任务委托给其父类加载器去完成。这一过程自下而上,逐层向上委托,直到最高层的启动类加载器(Bootstrap ClassLoader)。如果父类加载器能够成功加载这个类,那么子类加载器就直接使用父类加载器加载的结果；如果父类加载器无法加载(例如找不到对应的类),则子类加载器才会尝试自己去加载。双亲委派机制具有如下优点:

- 保证类的唯一性:通过委托机制,确保了无论哪个类加载器加载,最终都是由同一个加载器加载到内存中的,避免了因重复加载而导致的类定义不一致问题。
- 提高加载的效率:利用缓存机制,如果父类加载器已经加载过某个类,子类加载器就不需要再次加载。
- 增强安全性:核心类库由最顶端的启动类加载器加载,确保了这些类不会被篡改或替代,增强了系统的安全稳定。

除了标准的 Java 类加载器遵循双亲委派模型外,自定义类加载器可以通过覆盖 loadClass 方法来改变这种委托行为,以满足特殊的需求,如热部署、模块化系统或实现隔离性等,但这种做法需谨慎,因为它可能破坏双亲委派带来的安全保障。
