Java 虚拟机在执行 Java 程序的过程中会把它所管理的内存划分为若干个不同的数据区域。这些区域有各自的用途,以及创建和销毁的时间,有的区域随着虚拟机进程的启动而一直存在,有些区域则是依赖用户线程的启动和结束而建立和销毁。根据《Java 虚拟机规范》的规定,Java 虚拟机所管理的内存将会包括以下几个运行时数据区域:

![Java虚拟机运行时数据区](/assets/img/jvm/jvm-01.png 'Java虚拟机运行时数据区')

## 1.程序计数器

**JVM 中的程序计数器(Program Counter Register)是一个比较特殊的内存区域,每个线程都有独立的程序计数器(线程私有),是唯一一个没有规定任何 OutOfMemoryError(OOM)情况的区域。它主要用于存放下一条将要被执行的字节码指令的地址。在 Java 多线程环境下,每个线程都有自己的程序计数器,互不影响,这样设计是为了线程切换后能恢复正确的执行位置**。由于 Java 虚拟机的多线程是通过线程轮流切换、分配处理器执行时间的方式来实现的,在任何一个确定的时刻,一个处理器(对于多核处理器来说是一个内核)都只会执行一条线程中的指令。因此,为了线程切换后能恢复到正确的执行位置,每条线程都需要有一个独立的程序计数器,各条线程之间计数器互不影响,独立存储,这类内存区域为"线程私有"的内存。

## 2.Java 虚拟机栈

**与程序计数器一样,Java 虚拟机栈(Java Virtual Machine Stack)也是线程私有的,它的生命周期与线程相同。虚拟机栈描述的是 Java 方法执行的线程内存模型:每个方法被执行的时候,Java 虚拟机都会同步创建一个栈帧用于存储局部变量表、操作数栈、动态连接、方法出口等信息。每一个方法被调用直至执行完毕的过程,就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程**。

局部变量表存放了编译期可知的各种 Java 虚拟机基本数据类型(boolean、byte、char、short、int、float、long、double)、对象引用(reference 类型,它并不等同于对象本身,可能是一个指向对象起始地址的引用指针,也可能是指向一个代表对象的句柄或者其他与此对象相关的位置)和 returnAddress 类型(指向了一条字节码指令的地址)。这些数据类型在局部变量表中的存储空间以局部变量槽(Slot)来表示,其中 64 位长度的 long 和 double 类型的数据会占用两个变量槽,其余的数据类型只占用一个。局部变量表所需的内存空间在编译期间完成分配,当进入一个方法时,这个方法需要在栈帧中分配多大的局部变量空间是完全确定的,在方法运行期间不会改变局部变量表的大小。

## 3.本地方法栈

**本地方法栈与 Java 虚拟机栈在作用和功能上有相似之处,但主要服务于 Java 方法以外的代码,即本地方法(Native Methods)。本地方法是由非 Java 语言(如 C、C++)编写并使用 JNI(Java Native Interface)接口调用的函数,它们可以直接访问操作系统底层资源和硬件。与 Java 虚拟机栈一样,每个 Java 线程都拥有自己的本地方法栈。这意味着,当线程执行本地方法时,会在自己的本地方法栈中进行操作,与其他线程的本地方法栈隔离**。

## 4.Java 堆

**Java 堆(Heap)是 Java 虚拟机(JVM)内存管理的一个核心区域,用于存储几乎所有的 Java 对象实例和数组。它是线程共享的,在虚拟机启动时创建,意味着所有线程都可以访问堆上的对象。Java 堆是垃圾收集(Garbage Collection, GC)的主要区域,因此也被称作 GC 堆**。Java 堆的大小可以在 JVM 启动时通过命令行参数(如 -Xms 设置初始堆大小,-Xmx 设置最大堆大小)进行设置,并且在程序运行期间,堆的大小可以根据需要动态地扩展(直到达到最大设定值)或收缩(取决于实现)。新创建的对象实例通常会优先分配在新生代的 Eden 区(年轻代的一部分),经过几次 GC 存活下来的对象会被移动到老年代(Old Generation)。**垃圾收集器(GC)会定期扫描堆内存,回收不再使用的对象所占用的空间,以保证有足够的内存供新的对象分配。由于内存资源是有限的,当堆内存不足以满足新的内存分配请求时,JVM 会尝试触发垃圾回收来释放空间。如果回收后仍然没有足够的内存,或者达到了堆的最大限制,JVM 会抛出 OutOfMemoryError 异常,表明堆内存耗尽**。

### 4.1 堆内部结构

![JVM堆结构](/assets/img/jvm/jvm-03.png 'JVM堆结构')

JVM 采用堆空间分代思想将堆结构分为**新生代、老年代和元空间**三部分:

- **新生代(Young Generation)**:这是对象最初分配内存的地方,新生代由 Eden 区和 Survivor 组成:
  - **Eden 区**:大多数新创建(new)的对象首先分配在这里。
  - **Survivor 区**:分为两个相等大小的区域,通常称为 S0(Survivor 0)和 S1(Survivor 1)。在新生代垃圾回收(Minor GC)后,存活的对象会被移动到其中一个 Survivor 区,另一个用于下一次 GC 时使用。这两个区域交替充当“from”和“to”角色,帮助筛选出长期存活的对象。
- **老年代(Old Generation / Tenured Generation)**:当对象在新生代中经过几次垃圾回收仍然存活,它们会被移动到老年代。老年代存放寿命较长的对象,其大小通常比新生代大,垃圾回收频率较低,但每次回收耗时更长,如 Major GC 或 Full GC。
- **元空间(Metaspace)**:一个特殊区域,不属于堆内存的直接划分。在 Java 8 及以后版本中,永久代被元空间取代。元空间用于存储类的元数据信息,如类的结构信息等,它不在 JVM 的堆内存中,而是使用本地内存。这样做消除了永久代容易引发的内存溢出问题,并且元空间的大小是可以动态调整的。

堆空间分代思想是 Java 虚拟机(JVM)内存管理中的核心概念之一,旨在通过区分不同生命周期的对象来优化垃圾收集(Garbage Collection, GC)的性能:

- **优化 GC 性能**:通过区分对象的生命周期,JVM 可以在较小的、对象存活率低的新生代频繁执行快速的垃圾回收,而在较大的、对象存活率高的老年代执行较少但更耗时的回收。
- **减少 STW 时间**:分代收集可以针对性地回收那些包含大量可回收对象的区域,从而减少整个堆的冻结时间(Stop-The-World)。
- **提升内存分配效率**:新生代的小对象分配和回收更快,而老年代的大对象分配也更为直接,两者分开管理提高了整体的内存分配和回收效率。

### 4.2 设置堆内存大小

设置 Java 虚拟机(JVM)的堆内存大小可以通过在启动 Java 应用程序时,向 Java 命令行添加特定的 JVM 参数来实现。主要使用的参数是`-Xms`和`-Xmx`,分别用于设置初始堆大小和最大堆大小:

- -Xms:-Xms 表示设置堆空间的初始大小,等同于`-XX:InitialHeapSize`。
- -Xmx:-Xmx 表示设置堆空间的最大内存,等同于`-XX:MaxHeapSize`。

```shell
# 使用java -jar 命令启动hello.jar,指定堆空间初始内存为1g,堆空间最大内存为1g
java -Xms1g -Xmx1g -jar hello.jar

# 等同于上面的命令
java -XX:InitialHeapSize=1g -XX:MaxHeapSize=1g -jar hello.jar
```

一旦堆空间消耗的内存大于`-Xmx`指定的最大内存时,就会抛出堆内存溢出(OutOfMemoryError)异常。在实际开发中,通常会将 JVM 的`-Xms`(初始堆大小)和`-Xmx`(最大堆大小)设置为相同的值,原因如下:

- **减少运行时的内存分配延迟**:当-Xms 和-Xmx 不同,JVM 会在运行时根据需要动态调整堆内存大小,直到达到-Xmx 设定的最大值。这一动态调整过程可能导致额外的性能开销,因为每次扩展堆内存时,JVM 可能需要执行内存整理和压缩操作,以分配连续的内存空间,这会对应用性能产生负面影响。
- **避免垃圾收集频率的变化**:堆大小的动态变化会影响垃圾收集器的行为。较小的堆会导致更频繁的垃圾收集,而较大的堆则会延长两次垃圾收集之间的周期。这种不一致性可能导致应用性能波动。设置相同的-Xms 和-Xmx 可以让垃圾收集行为更加稳定,因为堆大小在应用生命周期内是固定的。
- **提高应用启动性能**:对于需要快速启动并迅速达到稳定状态的应用,一开始就分配全部预定的堆内存可以减少启动和初期运行时的内存分配延迟。这对于追求快速响应和启动时间的应用场景尤为重要。

如果不手动指定-Xms 和-Xmx,JVM 将会根据以下规则计算最小堆内存和最大堆内存:

- 最小堆内存:最小不得小于 8M,如果物理内存大于等于 1G,那么默认值为物理内存的 1/64,即 1024MB/64=16MB。最小堆内存会在 JVM 启动时被初始化。
- 最大堆内存:如果物理内存小于 192MB,那么最大堆内存的默认值为物理内存的一半;如果物理内存大于等于 1G,那么最大堆内存的默认值为物理内存的 1/4。

```java
public class HeapSize {
    public static void main(String[] args) {
        // 最小堆内存大小
        long initialMemory = Runtime.getRuntime().totalMemory() / 1024 / 1024;
        // 最大堆内存大小
        long maxMemory = Runtime.getRuntime().maxMemory() / 1024 / 1024;
        // 物理内存为32G
        System.out.println("-Xms:" + initialMemory + "MB"); // -Xms:512MB
        System.out.println("-Xmx:" + maxMemory + "MB"); // -Xmx:8156MB
    }
}
```

### Minor GC、Major GC、Full GC

在 HotSpot VM 实现中,根据作用的内存区域不同,垃圾回收可以大致分为 Partial GC 和 Full GC 两种类型,其中 Partial GC 表示部分 GC,包括 Minor GC、Major GC(有时也称为 Old GC)、Mixed GC 三种。Full GC 表示全局 GC(也叫整堆 GC),会针对堆中新生代、老年代、元空间三个区域进行 GC。

- **Minor GC(次要 GC)**:Minor GC 主要针对新生代的 Eden 区和 Survivor 区进行垃圾回收。由于新生代中的对象大多生命周期较短,因此 Minor GC 较为频繁,且通常执行速度较快。当新生代空间不足时,会触发 Minor GC。此过程会将 Eden 区和其中一个 Survivor 区中仍然存活的对象复制到另一个 Survivor 区或直接晋升到老年代。
- **Major GC(主要 GC)**:Major GC 主要清理老年代中的不再使用的对象。相比于 Minor GC,Major GC 涉及的对象数量更多,存活时间更长,因此其执行时间一般较长(比 Minor GC 慢 10 倍)。
  - 当出现 Major GC 时,通常会伴随着一次 Minor GC(并非绝对,在 ParallelScavenge 收集器的收集策略就有直接进行 Major GC 的过程)。
  - 当老年代的内存空间不足时,首先会尝试触发 Minor GC,如果 Minor GC 后内存空间仍不足,则触发 Major GC,如果 Major GC 后内存空间仍不足,则抛出 OOM 错误。
- **Mixed GC(混合 GC)**:收集整个新生代和部分老年代的垃圾收集。目前只有 G1 GC 会有这种行为。
- **Full GC**:是覆盖整个 Java 堆和方法区的完整垃圾回收过程。由于涉及整个堆内存(包括新生代、老年代以及元空间),因此 Full GC 是最为耗时的操作,容易引起应用程序暂停时间较长(STW,Stop-The-World)。因此,减少 Full GC 的发生频率对提升应用性能至关重要。Full GC 触发时机:
  - 老年代空间不足。
  - 元空间不足。
  - 系统显式调用 System.gc(),虽然这是一个建议而非强制执行,现代 JVM 通常会忽略这个调用以避免不必要的 Full GC。
  - 在某些垃圾回收器(如 CMS)的使用场景中,如果新生代晋升到老年代的对象平均大小超过老年代剩余空间,也可能触发 Full GC。
  - 由 Eden 区、Survivor space 0(From Space) 向 Survivor space 1(To Space)复制时,对象大小大于 To Space 可用内存,则会把该对象转存至老年代,且老年代的可用内存小于该对象大小。

:::tip
STW,全称 Stop-The-World,是 Java 虚拟机(JVM)在进行某些内部操作时的一种机制,它会导致整个 Java 应用暂时停止响应外部请求。当 STW 发生时,以下情况会出现:

- 所有应用线程暂停:无论是用户线程还是非用户线程,都会被 JVM 强制暂停。这意味着正在进行的所有任务都会暂时中断,包括数据处理、网络通信等。
- 仅允许特定操作:虽然 Java 代码执行停止,但某些 JVM 内部的守护线程或“VM Threads”(如垃圾回收线程)仍可以继续执行。此外,原生代码(native code)可以继续运行,但不能与 JVM 交互,因为 JVM 此时不响应任何外部请求。
- 严重影响应用程序性能:STW 对应用性能有显著影响,因为它导致应用暂停,增加响应时间,降低吞吐量。对于需要低延迟响应或实时处理的应用而言,频繁的 STW 是一个严重问题。

STW 是 JVM 为了执行某些维护任务(特别是垃圾回收)而采取的一种全局暂停策略,它体现了在追求自动内存管理便利性的同时,不得不面对的性能挑战。现代 JVM 和垃圾回收算法(如 G1、ZGC、Shenandoah)不断进化,致力于减少 STW 时间甚至实现几乎无停顿的垃圾回收,以提高应用的性能和用户体验。
:::

## 5.方法区

**JVM(Java 虚拟机)中的方法区(Method Area)是一个用于存储类结构信息(包括类的结构信息,如类的全限定名、访问修饰符、超类、接口、字段、方法等元数据)、常量池(存储编译期生成的各种字面量和符号引用,如字符串常量、类名、方法名等。运行时常量池是方法区的一部分)、静态变量(类的静态字段和静态 final 字段的存储区域)、即时编译器编译后的代码(JIT 编译器将热点代码编译成本地机器码后,会存储在这里,以提高程序执行效率)等数据的内存区域。方法区存放的数据虽然生命周期较长,但并非永不回收。类的卸载和常量池的清理是方法区垃圾回收的一部分,但这个过程比堆的垃圾回收要少得多,条件也更加苛刻**。

在不同的 JVM 版本和实现中,方法区有着不同的名称和特性,例如,在 Java 8 之前的 HotSpot JVM 中,它被称为永久代(PermGen space),而在 Java 8 及以后的版本中,永久代被元数据区(Metaspace)所替代,以解决永久代空间不足的问题。由于永久代有其固有的问题,如空间不足、内存泄漏风险、与 JVM 堆大小调整不灵活等,从 Java 8 开始,HotSpot JVM 引入了元数据区(Metaspace),它不再使用 JVM 堆的一部分,而是直接使用本机内存,这允许更灵活和动态的内存管理,减少了内存溢出的风险,并提高了性能。
