## Redis 为什么是高性能的?

- **纯内存操作**:Redis 的所有操作都在内存中完成,消除了磁盘 I/O 的巨大开销。磁盘寻道时间在毫秒(ms)级别，而内存访问在纳秒(ns)级别,性能相差万倍。
- **高效的数据结构**:Redis 并不是简单地存储 KV,它对每种数据类型都进行了极其严苛的内存和性能优化:
  - SDS (Simple Dynamic String)：相比 C 语言字符串，获取长度的时间复杂度为 `O(1)`,且能减少内存重分配。
  - 跳表 (SkipList)：让 ZSet 在查询和插入时拥有平衡树般的 `O(\log N)` 性能，但实现更简单。
  - 压缩列表 (ZipList/QuickList):在数据量小时,通过连续内存空间大幅提升 CPU 缓存命中率。
- **采用单线程模型**:Redis在执行命令时使用单线程模型,由于没有锁竞争(锁的竞争和释放会产生巨大的性能损耗)和线程之间的上下文切换,执行效率比多线程模型快很多,Redis 的瓶颈通常在于内存大小或网络带宽,而不是 CPU 计算能力。为了突破单线程在网络读写上的瓶颈,Redis 6.0 引入了 I/O Threads,并行化网络读写,将最耗时的“读取数据、解析协议”和“写回结果”交给辅助线程处理。
- **采用非阻塞 I/O 多路复用 (I/O Multiplexing)**:虽然核心处理是单线程,但对于连接处理 Redis 使用了基于 epoll 的 I/O 多路复用技术,一个线程同时监控数万个连接,只有当连接中有实际的命令发送过来时,Redis 才会去处理。
- **渐进式 Rehash**:渐进式 Rehash 是 Redis 在对哈希表(Dict)进行扩容或收缩时,不一次性搬迁所有键值对,而是分多次、渐进式地将 ht[0]（旧表）中的数据迁移到 ht[1]（新表）的过程。渐进式 Rehash 是 Redis 应对‘大数据量下高可用’的核心武器。它巧妙地利用了双哈希表结构,通过 rehashidx 计数器将搬迁压力线性化。在 Rehash 期间,它能同时兼顾新老数据表的读写请求,既避免了全量迁移导致的瞬时阻塞,又通过‘只增新表’的策略确保了迁移最终能收敛完成。

## Redis 的基础数据结构有哪些?

Redis 支持多种基础数据结构,基础数据结构包括 String(字符串)、List(列表)、Set(集合)、Sorted Set(有序集合)、Hash(哈希表),除了基础数据结构外,Redis 还提供了 Bitmap、Stream、HyperLogLog、Geospatial 等高级数据结构。

## Redis 的应用场景有哪些?

- 分布式缓存:在分布式系统中,Redis 可以被用作全局缓存,提供共享数据和信息。在实际项目中,为了进一步提升应用性能,应用中可能存在多级缓存,例如本地缓存、分布式缓存等。当进行读操作时,为了减少网络 IO 和带宽,优先访问本地缓存,如果命中缓存则立即返回,如果未命中则从分布式缓存中查找,命中缓存则立即返回,否则将访问数据源并将执行结果填充至多级缓存中。
- 分布式锁: Redis 可以用作分布式锁的实现,通过 SETNX 命令来实现简单的锁机制,避免多个节点同时执行某个关键任务。使用 Redis 实现分布式锁需注意如下事项:
  - 互斥:同一时刻保证只有一个线程获取到锁,其他线程阻塞等待获取锁。
  - 锁应设置超时时间:在分布式环境下,由于 Redis 故障或网络 IO 等原因导致锁持有方无法及时释放锁资源,其他节点则会长时间排队阻塞等待获取锁资源,可能出现死锁现象,严重影响程序执行效率。为了防止释放锁失败,通常会设置锁的超时时间,一旦锁长时间未被释放,则锁自动过期,释放锁后其他线程竞争锁。
  - 锁的可重入性:锁的重入性是指同一个线程是否可以多次获取同一把锁而不被阻塞
  - 锁的续期:如果已经设置了锁的过期时间,当到锁过期但是业务逻辑还未执行完成,此时无法保证业务正常执行,在这种情况应需要支持锁的自动续期。
- 分布式队列: Redis 的发布/订阅机制和 List 结构、Stream 结构可以用来构建轻量级的消息队列。生产者将消息发布到 Redis,而消费者则订阅这些消息。这种机制可以用于实现异步任务、事件通知等。Redis 实现分布式队列可能会存在如下问题:
  - 无法保证可靠性:Redis 的设计目标是提供高性能、低延迟的分布式系统,无法保证强一致性和可靠性,某些情况下,Redis 会丢失消息,或者在出现故障时无法提供持久化保证。
  - 缺少 MQ 的高级特性:相较于专业级 MQ,基于 Redis 实现分布式队列虽然简单,但是不支持 MQ 的高级特性,例如消息重试、消息路由、持久化消息等。
- 分布式会话:Redis 可以用于存储用户会话数据,特别是在分布式系统中。将用户的会话信息存储在 Redis 中,可以实现快速、可扩展、可靠的用户会话管理。
- 排行榜和计分系统:利用有序集合和 Redis 提供的排名功能,可以很方便地构建排行榜和计分系统,例如游戏中的玩家排名。
- 计数器应用:Redis 支持原子性的增减操作,因此可以用来实现计数器应用。例如,可以用 Redis 记录网站的访问次数、文章的点赞数等。
- 实时分析:Redis 支持复杂的数据结构,如 HyperLogLog 和布隆过滤器,可用于实时分析场景,例如统计在线用户数、IP 去重等。
- 地理位置服务:Redis 提供了有序集合(Sorted Set)的支持,可用于存储和查询地理位置信息。这对于构建位置服务应用如地理位置推荐、附近的人等非常有用。

## Redis 的数据过期策略?

当数据过期时,Redis为了性能与内存的最优平衡,并不会立马删除数据,而是采用惰性删除 + 定期采样删除的混合模式来保证CPU吞吐量和内存压力。Redis 主要以下三种数据过期策略:

- 惰性删除:当 Key 过期后,Redis 不会自动去检查它。只有当客户端尝试访问(GET/SET 等)这个 Key 时,Redis 才会检查它是否过期。惰性删除虽然可以减少CPU操作频次,如果大量过期的 Key 未被访问,它们会常驻在内存中无法释放,造成内存泄漏。
- 定期删除:为了弥补惰性删除的缺陷,Redis 默认每隔 100ms(由配置项 hz 决定)运行一次定时任务,随机抽取一部分设置了过期时间的 Key 进行检查
  并删除,如果已过期 Key 的比例超过 25%则重复检查。定期删除通过限制清理时长和频率,将 CPU 消耗控制在可接受的范围内,同时有效释放内存,可以有效避免内存泄漏。
- 内存淘汰策略:如果过期 Key 还没来得及删,内存就已经满了(达到 maxmemory),Redis会触发内存淘汰策略,根据策略删除已过期的数据。

## Redis 内存淘汰策略?

由于 Redis 一种内存数据库,所有数据都存储在内存中。如果不进行任何限制和管理,内存使用量会不断增加,最终会导致系统内存耗尽,影响系统的稳定性和性能。因此,内存淘汰策略在 Redis 中扮演着重要的角色, 用于在达到内存限制时,决定哪些数据需要被移除,以便腾出空间给新数据。

Redis 提供了多种内存淘汰策略,可以通过配置文件或命令行参数进行设置。这些策略分为删除策略和淘汰策略两类:

- 删除策略:删除策略决定在没有新数据写入时,如何删除过期的数据:
  - noeviction:当内存达到限制时,不会再接受写入请求。客户端会收到错误信息。
  - allkeys-lru:在所有的键中,移除最近最少使用的键(LRU:Least Recently Used)。
  - volatile-lru:在设置了过期时间的键中,移除最近最少使用的键。
  - allkeys-random:在所有的键中,随机移除某个键。
  - volatile-random:在设置了过期时间的键中,随机移除某个键。
  - volatile-ttl:在设置了过期时间的键中,移除剩余时间最短的键。

- 内存淘汰策略:
  - noeviction:当内存使用达到限制时,不允许任何新的写入操作,包括插入新键和修改已有键。客户端会收到一个错误信息。这种策略适用于需要严格控制数据集大小,且不希望自动移除数据的场景。
  - allkeys-lru:在所有键中移除最近最少使用的键。LRU 算法会记录每个键的访问时间,当需要移除键时,会选择访问时间最久的键。这种策略适用于需要缓存大量数据,且希望根据访问频率动态调整缓存内容的场景。
  - volatile-lru:只在设置了过期时间的键中移除最近最少使用的键。这种策略适用于希望缓存的数据自动过期,但仍希望根据访问频率调整缓存内容的场景。
  - allkeys-random:在所有键中随机选择一个键进行移除。这种策略适用于不希望依赖访问频率,只希望简单随机移除数据的场景。
  - volatile-random:只在设置了过期时间的键中随机选择一个键进行移除。这种策略适用于希望缓存的数据自动过期,并且希望简单随机移除数据的场景。
  - volatile-ttl:只在设置了过期时间的键中移除剩余时间最短的键。这种策略适用于希望根据剩余存活时间调整缓存内容的场景。

通过合理选择和配置内存淘汰策略,可以有效管理 Redis 的内存使用,确保系统的稳定性和性能:

- 缓存频繁访问的数据:allkeys-lru 或 volatile-lru 是常用的选择,因为它们基于访问频率进行数据淘汰,能有效保持热数据在缓存中。
- 严格控制数据集大小:noeviction 可以确保数据不会被自动移除,但需要注意可能会导致写入操作失败。
- 随机淘汰数据:allkeys-random 或 volatile-random 适用于简单随机淘汰数据的场景。
- 基于数据的存活时间淘汰:volatile-ttl 适用于希望根据剩余存活时间进行数据淘汰的场景。

## Redis 的持久化机制?

Redis 是一个内存数据库,但它提供了多种持久化机制来确保数据在内存意外丢失(如系统崩溃或重启)后可以恢复。Redis 的主要持久化机制有 RDB(Redis DataBase)、AOF(Append Only File)、RDB 和 AOF 混合模式。

### RDB

RDB 持久化是指将某一时刻的数据快照保存到磁盘中。Redis 会在指定的时间间隔内生成数据的快照,并将其保存到 .rdb 文件中。其缺点如下:

- **数据加载速度快**:RDB 文件是紧凑的二进制文件,因此加载速度非常快。
- **对读性能影响小**:在生成快照时,父进程会 fork(分叉)一个子进程来处理,子进程在完成后将数据写入磁盘,这样减少了对主进程读写操作的影响。
- **数据不够实时**:RDB 持久化是定时生成快照的,如果系统在两个快照之间崩溃,所有在此期间的更改都将丢失。
- **对写性能有一定影响**:生成快照时会 fork 一个子进程,这个操作在数据量很大时可能会影响性能。

redis.conf 中配置 RDB 持久化例子:

```shell
save 900 1   # 900秒(15分钟)内如果至少有1个key被改变,则触发持久化
save 300 10  # 300秒(5分钟)内如果至少有10个key被改变,则触发持久化
save 60 10000 # 60秒(1分钟)内如果至少有10000个key被改变,则触发持久化
```

### AOF 模式

AOF 持久化记录了每次对数据库的写操作,将其以日志的形式追加到文件中。Redis 可以通过重新执行这些命令来重建数据库。优缺点如下:

- **更高的数据安全性**:AOF 通过追加写命令日志的方式,可以实现更高的实时性和数据持久化,最大程度地避免数据丢失。
- **可调整的持久化频率**:可以配置写入磁盘的频率,如每秒一次(appendfsync everysec),每次写操作都同步到磁盘(appendfsync always),或让操作系统决定何时写入磁盘(appendfsync no)。
- **文件体积大**:相比 RDB 文件,AOF 文件通常更大,因为它记录了每一个写操作。
- **数据恢复速度较慢**:AOF 文件需要重放所有的写命令,这使得数据恢复的时间较长。
- **对写性能有影响**:因为每次写操作都需要追加到 AOF 文件中,频繁的磁盘 I/O 操作会影响性能。

redis.conf 中配置 AOF 持久化例子:

```shell
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec   # 每秒同步一次(推荐)
# appendfsync always    # 每次写操作都同步
# appendfsync no        # 不同步,由操作系统决定何时写入
```

### RDB 和 AOF 混合模式

Redis 4.0 引入了 RDB 和 AOF 的混合持久化模式,结合了两者的优点,既能快速恢复数据,又能减少数据丢失。在这种模式下,Redis 会定期生成 RDB 快照,并在 RDB 文件的末尾追加 AOF 日志。

redis.conf 中配置 RDB 和 AOF 的混合持久化例子:

```shell
# 启用RDB和AOF混合模式
aof-use-rdb-preamble yes
```

## 什么是 Redis Pipeline?

Redis Pipeline 是一种通过批量发送命令来减少网络往返次数的技术。它允许客户端在一次网络请求中发送多个命令,然后一次性读取所有命令的响应。这样可以显著提高性能,特别是在需要执行大量命令的情况下。Pipeline 优点:

- 减少网络延迟:在普通的请求-响应模式下,每个命令都需要经过一次网络往返,而 Pipeline 技术允许多个命令在一个网络请求中发送,从而减少了网络往返次数。
- 提高吞吐量:由于减少了每个命令的网络延迟,Redis 服务器可以更快地处理更多的命令,整体性能显著提升。
- 降低网络开销:批量发送命令减少了 TCP 包的数量,从而降低了网络开销,提高了带宽利用率。

Pipeline 适用于以下场景:

- 批量操作:Pipeline 最常用于批量操作,即在一次网络请求中执行大量命令。例如,批量插入、批量删除、批量更新等操作都可以通过 Pipeline 来优化。
- LUA 脚本:Lua 脚本(Redis Scripting)和 Pipeline 搭配使用可以进一步优化性能。Lua 脚本允许在 Redis 服务器端执行复杂的业务逻辑,减少客户端与服务器之间的网络往返次数。将批量操作封装在一个 Lua 脚本中,可以在服务器端原子性地执行这些操作。
- 事务(MULTI/EXEC):Redis 事务(MULTI/EXEC)和 Pipeline 可以结合使用,以确保批量命令的原子性执行。通过使用 MULTI 命令开启事务,并在 EXEC 命令中提交事务,可以保证所有命令要么全部执行,要么全部不执行。

使用 Pipeline 注意事项:

- 内存消耗:由于 Pipeline 会将所有命令和响应保存在内存中,如果命令数量过多,可能会导致客户端或服务器内存消耗过大。
- 命令顺序:在 Pipeline 中,命令的执行顺序和发送顺序是一致的,因此需要注意命令之间的依赖关系。
- 错误处理:在 Pipeline 中,如果某个命令出错,其他命令仍会执行,因此需要在客户端处理响应中的错误。

## Redis 的集群模式有哪几种?

Redis 提供了主从复制、哨兵模式、Cluster(集群)模式来实现数据的高可用性和扩展性。

### Redis 主从复制

主从复制模式是 Redis 最基本的集群模式,通过将数据从一个主节点(Master)复制到一个或多个从节点(Slave)来实现高可用性和读写分离。其特点如下:

- 支持高可用:如果主节点故障,从节点可以接管,提供读写服务。
- 读写分离:主节点负责写操作,从节点负责读操作,提高系统的读性能。
- 手动故障转移:需要手动将从节点提升为主节点。

由于主从复制模式配置简单,读性能高,但是主节点发生单点故障,需要手动故障转移,操作比较复杂,适用于小规模数据集,要求简单高效的读写分离,手动管理故障转移的场景。主从复制工作流程如下:

- 初次同步:
  - 从节点连接到主节点并发送 SYNC 命令。
  - 主节点接收到 SYNC 命令后,会执行一个后台快照(RDB 快照)并将快照文件传送给从节点。
  - 同时,主节点将新的写命令存储在内存缓冲区中。
  - 从节点接收到快照文件后,清空自身数据并载入快照数据。
  - 之后,主节点将缓冲区中的写命令发送给从节点,从节点应用这些写命令,完成数据同步。
- 持续同步:
  - 初次同步完成后,主节点将每个写命令都发送给从节点。
  - 从节点接收到命令后,立即执行,以确保数据的一致性。

主从节点同步数据时,由于网络延迟或主从节点负载过高等原因会导致主从数据同步延迟,会出现短暂的数据不一致性问题。

### Sentinel(哨兵)模式模式

Sentinel 是 Redis 提供的高可用解决方案,通过监控主从结构的节点状态,在主节点故障时自动进行故障转移。其特点如下:

- 支持自动故障转移:哨兵节点监控主从节点状态,自动将从节点提升为主节点。在 Redis Sentinel 模式下,官方推荐最小配置是至少三台 Sentinel 服务器(还需要一主一从),三台 Sentinel 服务器可以确保在任意一台 Sentinel 服务器失效的情况下,依然可以达成多数(quorum)来决定故障转移,以保证高可用性和防止脑裂(在分布式系统中,脑裂是指网络分区导致集群被分为两个或多个部分,各部分可能会各自选出新的主节点,导致数据不一致)。
- 配置发现:客户端可以通过哨兵获取当前主节点的地址,实现自动化配置。2.MQ 常见问题
- 高可用性:多哨兵监控确保高可用性。

Sentinel 模式下,配置和运维复杂度较高,需要部署和管理哨兵节点,通常适用于中小规模数据集,要求高可用性和自动故障转移,能接受一定的管理复杂度。

### Redis Cluster 模式

Redis Cluster 是 Redis 的分布式解决方案,通过将数据分片(sharding)存储在多个节点上,实现数据的水平扩展和高可用性。其特点如下:

- 支持数据分片:将数据分成多个哈希槽(hash slots),每个节点负责一定数量的哈希槽。
- 支持自动故障转移:集群模式下自动进行故障转移,确保高可用性。
- 无中心架构:没有单点故障,所有节点彼此连接,互相感知。

Cluster 模式下,支持水平扩展,高可用性,自动故障转移,但是配置和管理较为复杂,客户端需要支持 Redis Cluster 协议。适用于大规模数据集,要求水平扩展和高可用性,能接受较高的配置和管理复杂度。

## 什么是Redis集群脑裂问题?

Redis脑裂是指由于**网络分区（Network Partition）**导致集群中同时出现了两个 Master 节点。例如Master 节点与 Slave 节点、Sentinel（哨兵）之间的网络突然断开,但 Master 与客户端(Client)之间的网络是通的,由于联系不上 Master，哨兵会发起故障转移(Failover),将其中一个 Slave 提升为新 Master。此时一部分客户端可能还在向旧 Master 写入数据,当网络恢复,旧 Master 会被降级为 Slave,并从新 Master 同步数据。由于 Slave 同步是先清空自己,旧 Master 在脑裂期间接收的所有写入数据都会彻底丢失。

脑裂的本质是局部视图不一致(分布式共识的延迟),对于旧 Master,认为自己还活着,继续处理请求;对于哨兵集群,认为旧 Master 已死,选出新主。要解决脑裂,核心思路是:限制 Master 在异常情况下的写入能力。Redis 提供了两个关键配置参数:

```txt
# 1. Master 必须拥有至少多少个 Slave 才能进行写操作
min-replicas-to-write 1

# 2. Slave 与 Master 通信的延迟阈值（秒）
# 如果 Slave 在 10 秒内没给我回消息，我就认为这个 Slave 失效了
min-replicas-max-lag 10
```

工作流程如下:

- 当发生网络分区,旧 Master 发现自己联系不到足够数量的 Slave(或者延迟超过 10s)。
- 旧 Master 会自动拒绝客户端的写请求（返回错误）。
- 这样,即便它还顶着 Master 的头衔,也不会产生新的脏数据。
- 当网络恢复,它作为 Slave 挂载到新 Master 下时,丢失的只是“拒绝写入”期间的数据,而不是“成功写入后被覆盖”的数据。

如果脑裂已经发生且导致了数据丢失，通常有以下处理方式：

- 应用层双写/补偿:在业务代码中记录写入日志,对比新旧实例的 AOF 文件进行手动补录(成本极高)。
- 监控报警:监控 slave0: state=online 的数量,一旦触发 min-replicas 限制,立即介入。

简单来说,Redis 脑裂本质上是 CAP 定理中 分区容错性(P) 与 一致性（C） 的权衡问题。Redis 默认偏向可用性(AP),因此在极端网络分区下存在丢失数据的风险。在金融或库存等高一致性场景中,可以通过配置 `min-replicas-to-write` 强制将 Redis 转化为类似 CP 的模式,牺牲一部分可用性(在 Slave 不足时拒绝写入)来换取数据的最终确定性。同时,配合哨兵的 quorum 配置,保故障转移的合法性,从闭环上降低脑裂影响。

## 为什么 Redis Cluster 槽位是 16384?

简单来说,Redis Cluster Slot 选择 `16384 ($2^{14}$)` 是在 消息大小、集群规模和压缩效率之间做出的最优权衡,虽然 16384 刚好是 `2^{14}`,但其实 Redis 并不强求它是 2 的幂次方,在定位 Key 属于哪个槽位时,Redis 使用的是 `CRC16(key) % 16384`,2的幂次方位运算的效率更高。

- 减少消息头的大小:在 Redis Cluster 模式下,节点之间会频繁地交换 Gossip 消息(心跳包),用于同步集群状态。每个心跳包的头部都会携带一份自己的 slots 槽位图,如果槽位是 `16384`,位图大小为：`16384 除以 8 = 2048` 字节(即 2KB)。如果槽位增加到 `65536 (2^{16})`,位图大小将飙升至：`65536 除以 8 = 8192` 字节(即 8KB)。
- 集群规模的限制:Redis Cluster 的设计目标并不是支持无限扩展,官方建议的集群最大规模通常在 1000 个节点左右。对于 1000 个节点来说，16384 个槽位已经足够让每个节点平均分到 16 个左右的槽位,能够实现非常均匀的数据分布。增加到 65536 个槽位并不能给 1000 个以内的节点带来明显的分布式优势,反而增加了网络负担。
- 提升位图压缩效率:Redis 在传输槽位图时会尝试进行压缩。位图(Bitmap)的填充率如果过高,压缩效果就很差。在节点数较少(如少于 1000)的情况下,使用 `2^{14}` 的位图,其空间利用率和压缩算法的匹配度最高。

Redis Cluster 选择 16384 个槽位是一个基于工程实践的权衡(Trade-off)。虽然理论上 65536 槽位能支持更多节点,但在 Redis 建议的千级节点规模内,16384 既能保证数据分布的均匀性,又能将 Gossip 协议的心跳包头部控制在 2KB 这一平衡点。这避免了因槽位图过大导致的带宽过度占用,是针对大规模分布式缓存场景下的 性能与通信成本 的最优解。
