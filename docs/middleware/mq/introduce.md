## 1.MQ 介绍

MQ 是 Message Queue(消息队列)的简称,队列是一种先进先出的数据结构,队列当中存储的就是消息,消息的类型可以是文本字符串、JSON、内嵌对象、byte。消费者可以消费消息,消费一个消息队列就会删除一个消息,提供者可以提供消息,提供一个消息队列就会增加一个消息。当大量消息存储在队列当中且未被消费时,会造成大量消息堆积。MQ 常用于以下场景:

- **应用解耦**:在项目启动之初来预测将来会碰到什么需求是极其困难的。消息中间件在处理过程中间插入了一个隐含的、基于数据的接口层,两边的处理过程都要实现这一接口,这允许你独立地扩展或修改两边的处理过程,只要确保它们遵守同样的接口约束即可。
- **流量削峰填谷**:在访问量剧增的情况下,应用仍然需要继续发挥作用,但是这样的突发流量并不常见。如果以能处理这类峰值为标准而投入资源,无疑是巨大的浪费。使用消息中间件能够使关键组件支撑突发访问压力,不会因为突发的超负荷请求而完全崩溃。
- **异步通信**:在很多时候应用不想也不需要立即处理消息。消息中间件提供了异步处理机制,允许应用把一些消息放入消息中间件中,但并不立即处理它,在之后需要的时候再慢慢处理。

## MQ消费模型
在MQ中,"推"(Push)和"拉"(Pull)是两种基本的消息传送模型,它们描述了消息如何从消息队列传输到消费者(接收方)的过程。
- **Push(推)模型**:在推模型中,消息队列主动将消息推送给消费者。这意味着,当消息队列中有新消息时,它会自动将消息发送到与之关联的消费者。这种模式下,消费者需要事先订阅或者注册到特定的消息主题上,一旦有新消息发布,消息队列服务端就会立即将消息推送到所有订阅的客户端。Push(推)模型特点:
    - 适用于实时性要求较高的场景,消息产生后能迅速被消费者处理。
    - 对消费者来说实现简单,不需要不断地检查是否有新消息。消费者被动接受消息,如果消费速度跟不上推送速度,可能会导致消息积压或丢失。
- **Pull(拉)模型**:拉模型则是消费者主动从消息队列中拉取消息。在这种模式下,消费者按照自己的节奏,定期向消息队列请求消息。只有当消费者请求时,消息队列才会返回消息给消费者。消费者需要实现逻辑来不断检查消息队列是否有新消息可供处理。Pull(拉)模型特点:
    - 由于消费者需要根据时间间隔轮询拉取消息,因此实时性相比推模型要低。
    - 消费者可以根据自身处理能力控制消息消费的速度,灵活性更好,可以避免消息积压。
- **推拉混合模式**:推拉混合模式旨在融合推和拉两种模式的优点,是要求实时性和灵活性的一种平衡方式。例如在可以对高优先级或时间敏感的消息采用推模式,确保这些消息能被即时处理。对于非紧急或批量处理的消息,则可以使用拉模式,消费者根据自身负载情况适时拉取。混合模式还可以根据消费者的实时负载自动在推和拉之间切换,或者在系统低峰时段采用拉模式减少资源消耗,高峰时段转为推模式保证消息处理的及时性。

## 2.主流MQ对比

- RabbitMQ:RabbitMQ是一个基于Erlang实现的开源消息中间件系统,支持AMQP协议,具有并发能力强大,延时低(微秒级),性能优秀,适用于并发量一般(十万级至百万级)但对延迟敏感等场景。
- RocketMQ:RocketMQ是一个开源的分布式消息中间件,具有高吞吐量、低延迟、海量消息堆积、支持高级消息(顺序消息、事务消息、定时消息与延时消息)等特点,适用于电商、金融。
- Kafka:Kafka是一种分布式的流处理平台,特别适合处理高吞吐量的实时数据流。它不仅作为消息队列使用,还支持数据持久化和实时分析。具有极高的吞吐量,可扩展性强,支持分区和副本机制,保证数据的高可用性和持久性等特点,非常适合大数据处理(例如实时数据计算、日志采集)等领域。
